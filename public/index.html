Das ultimative Kontrollzentrum für dein Homelab

ModuLab.app ist ein hochperformanter Core, der Module als sicher sandboxed Plugins direkt in dein Dashboard integriert – ganz ohne eigenen Container pro Modul. Vollständig konfiguriert über eine moderne Weboberfläche, geschützt durch Verschlüsselung auf jeder Ebene und nahtlos integriert mit deinem OIDC-Provider.

Plattform-Funktionen

Ein moderner Kern für flexible Netze

Alles, was ein Homelab-Dashboard mitbringen muss – von Identität über Mehrsprachigkeit bis zur Performance.

🔑

OIDC SSO & Rollenmodell

Nutze deinen bevorzugten IdP (PocketID empfohlen, Authentik, Keycloak, Authelia oder jeder OIDC-konforme Provider). Ein Dynamic-Prefix-Gate kombiniert dein Präfix mit drei Gruppen – _Super-Admin, _Org-Admin, _User – für eine strikte, dreistufige Einlasskontrolle.

👥

Onboarding-Warteschleife

Automatische Profil-Synchronisation beim Erst-Login (Name, E-Mail, Avatar) per OIDC-Claims. Jeder neue Nutzer muss von einem Admin manuell freigeschaltet und einem Standort zugewiesen werden, bevor er Zugriff erhält.

🌐

Multi-Standort-Scoping

Verwalte physisch getrennte Netze (Zuhause, VPS/Cloud, Backup-Hosts) zentral. Per Row-Level-Security wird jeder Standort auf Datenbankebene isoliert – einzelne Module können zusätzlich gezielt standortübergreifend arbeiten (z. B. Netzwerk-Integrationen).

📦

Modul-Store & GitHub-Anbindung

Entdecke und installiere Module aus dem offiziellen Repository oder dem kuratierten Community-Index – mit Suche, Kategorie- und Tier-Filtern. Auch direkte ZIP-URLs (z. B. interne Repositories) werden unterstützt.

🔔

Echtzeit-Events

Server-Sent Events, verteilt über Valkey Pub/Sub, halten UI und Audit-Log live synchron – neue Logins, Modul-Status-Wechsel, Updates, Job-Fehler. Gekoppelt mit einer asynchronen Mail-Warteschlange für Benachrichtigungen.

⚡

Caching & Performance

Dreistufiges Valkey-Caching, PgBouncer-Connection-Pooling und lazy geladene Daten via TanStack Query sorgen für niedrige Latenz – auch auf schwächerer Homelab-Hardware. Ziel für die Startseite: First Contentful Paint < 300 ms im LAN.

🌍

Vollständige Mehrsprachigkeit

Jeder Text – UI, Fehlermeldungen, E-Mails – läuft über i18next. Englisch und Deutsch sind zu 100 % gepflegt; weitere Sprachen kann die Community per Pull Request beitragen.

♿

Barrierefreiheit nach WCAG 2.1 AA

Vollständige Tastaturnavigation, Screenreader-Labels, sichtbare Fokusringe und geprüfte Farbkontraste – automatisiert abgesichert durch axe-core in der CI-Pipeline.

📡

Offline-First

Dashboard, Auth und alle installierten Module funktionieren vollständig ohne Internetzugang. Nur der Modul-Store für neue Installationen und das Zertifikats-Renewal benötigen kurzzeitig eine Verbindung.

Plugin-Architektur

Module, die sich wie Core anfühlen – nicht wie Fremdkörper

Module laufen als Plugins, orchestriert vom Core, und nutzen dessen Infrastruktur direkt – eigenes PostgreSQL-Schema, gemeinsames Caching, gemeinsame Auth. Kein eigener Container, kein Reverse-Proxy-Gefühl: die Modul-UI rendert nativ im Dashboard, technisch isoliert durch einen sandboxed iFrame.

Stufe 1

Config-driven

Reine YAML-Deklaration im manifest.yaml – kein eigener Code. Core generiert CRUD-Endpunkte und optional sogar die UI automatisch.

Beispiel: Rezepte-Verwaltung

Stufe 2

TypeScript-Handler

Eigene Geschäftslogik als TypeScript-Handler, ausgeführt in einem isolierten Deno-Worker mit eigenem Timeout und Memory-Cap.

Beispiel: Haushaltsbuch, internes Wiki

Stufe 3

Wie Stufe 2, zusätzlich mit deklarierten, fest erlaubten externen Verbindungen (egress_allowlist) und typisierten Credentials.

Beispiel: UniFi RADIUS, Cloudflare

Zero-Trust Homelab

Sicherheit auf jeder Ebene

Vom Edge-Router bis zum einzelnen Modul-Aufruf – jede Schicht des Systems wurde explizit gehärtet.

Netzwerk & Edge

✓

Zero-Expose-Policy: Außer Traefik (Port 443) ist kein einziger Container nach außen erreichbar. Header-Sanitization verhindert Request-Smuggling.

✓

Rate-Limiting & Circuit-Breaker: Konfigurierbar pro Route/Modul, automatische Isolation bei Layer-7-Anomalien (DDoS, Error-Spikes).

✓

CSRF-Eliminierung: Konsequente Bearer-Token-Strategie statt Cookies – kein Browser sendet den Authorization-Header automatisch mit, kein CSRF-Token-Mechanismus nötig.

✓

Strikte CSP: frame-src 'self' erlaubt ausschließlich Core-eigene Modul-Bundles, jede externe Drittseite wird blockiert.

Verschlüsselung

✓

Encrypt Everything: Sämtliche sensitiven Daten (Tokens, Passwörter, Modul-Credentials) liegen AES-256-GCM-verschlüsselt in der Datenbank.

✓

AES-256-SIV für indexierbare Felder: Auch durchsuchbare Daten bleiben verschlüsselt, ohne die Abfragefähigkeit zu verlieren.

✓

Key Management: Master-Key via Docker Secrets oder optional Infisical – ohne ihn bleibt die Datenbank unlesbarer Zeichensalat.

✓

Credential-Handling: Modul-Zugangsdaten werden entschlüsselt nur als typisierte Parameter übergeben – nie als Klartext-ENV, nie dem Frontend-Bundle bekannt.

Modul-Isolation

✓

Datenbank: Jedes Modul erhält ein eigenes PostgreSQL-Schema und eine eigene Rolle – physischer Zugriff auf fremde Schemas ist unmöglich, Row-Level-Security erzwingt zusätzlich die Standort-Isolation.

✓

Netzwerk: Deno --allow-net erlaubt exakt die in egress_allowlist deklarierten Hosts – kein Wildcard, kein genereller Internetzugriff.

✓

Dateisystem & Abhängigkeiten: Zugriff nur auf das eigene Storage-Verzeichnis; --cached-only erzwingt vorab vendorte Pakete, kein Laufzeit-Fetch.

✓

Prozess & Worker: Jedes Modul läuft in einem eigenen Deno-Worker mit Timeout/Memory-Cap; ein Circuit-Breaker isoliert wiederholt abstürzende Module automatisch nach 3 Abbrüchen in 10 Minuten.

Audit-Log & Supply-Chain

✓

Unveränderliches Audit-Log: PostgreSQL-Trigger blockieren UPDATE/DELETE (WORM); ein HMAC-SHA256-Chain-Hash macht nachträgliche Manipulation kryptografisch nachweisbar.

✓

Namespace-Enforcement: Module müssen dem modulab-mod-*-Namensschema folgen – schützt vor Typosquatting gegen Core-Komponenten.

✓

Checksumme & Signatur: SHA-256-Verifikation vor jeder Installation; offizielle Module sind zusätzlich per Cosign-Blob-Signing verpflichtend signiert, Community-Module transparent als signiert/unsigniert markiert.

✓

Bundle-Analyse & Lizenzpflicht: Statische Prüfung auf verdächtige Muster (Obfuskation, eval); ein SPDX-Lizenzfeld ist für jedes Modul Pflicht.

Häufig gestellte Fragen

Fragen zu ModuLab

Alles, was du über Lizenzierung, Sicherheit, OIDC und die Installation wissen musst.

Warum wurde die AGPLv3-Lizenz gewählt?

▼

Die AGPLv3 (GNU Affero General Public License v3) stellt sicher, dass der Quellcode des ModuLab-Cores für immer open-source und frei bleibt. Sie schützt das Projekt vor "SaaS-Trittbrettfahrern". Das bedeutet: Wer ModuLab modifiziert als gehosteten Netzwerk-Dienst (z. B. in der Cloud) anbietet, ist gesetzlich verpflichtet, seine Modifikationen ebenfalls als AGPLv3 offenzulegen. Community-Module können davon unabhängig eigene Lizenzen tragen.

Welche Identity Provider (IdPs) werden unterstützt?

▼

Da ModuLab vollständig auf dem standardisierten OpenID Connect (OIDC) Protokoll aufbaut, funktioniert es herstellerunabhängig mit jedem kompatiblen OIDC Identity Provider. Erfolgreich spezifiziert sind lokale Setups wie Authelia, Authentik, Keycloak und PocketID (empfohlen). Über standardisierte Claims werden Benutzername, Vorname, Nachname, E-Mail-Adresse und die Avatar-Bild-URL automatisch synchronisiert.

Gibt es bereits fertige Module für den Start?

▼

Nein, bewusst nicht. Version 1.0 liefert die Plattform selbst: ein vollständig funktionsfähiges Dashboard plus eine funktionierende GitHub-Anbindung zur Modul-Discovery und -Installation – auch wenn zum Launch noch kein einziges Modul existiert. Erst sobald dieses Fundament zu 100 % steht, beginnt die Entwicklung der eigentlichen Module über das offizielle modulab-module-sdk.

Kann ich ModuLab auch ohne Internetverbindung nutzen?

▼

Absolut. Solange dein OIDC-Provider lokal erreichbar ist, läuft das gesamte System vollständig offline. Die GitHub-API für den Modul-Store ist das einzige Feature, das eine Internetverbindung benötigt, um neue Updates oder Repositories abzufragen. Installierte Module arbeiten danach komplett autark in deinem Intranet.

Wie wird verhindert, dass ein fehlerhaftes Modul den ganzen Core lahmlegt?

▼

Jedes Modul mit eigenem Code läuft in einem dedizierten Deno-Worker mit Zeit- und Speicherlimit. Stürzt ein Worker ab, bleibt der Rest des Systems unberührt; nach drei Abstürzen in zehn Minuten isoliert sich das Modul automatisch (Status "isolated") und ein Admin wird per Mail/SSE alarmiert. Core selbst läuft als separater Go-Prozess und bleibt davon vollständig unbeeinflusst.

Wie steht es um Datenschutz/DSGVO?

▼

Da ModuLab selbst gehostet wird, liegen alle Daten ausschließlich auf deiner eigenen Infrastruktur. Betroffenenrechte sind dennoch direkt eingebaut: Nutzer können ihre Daten per Self-Service-Export abrufen, und eine Löschung erfolgt über Pseudonymisierung im unveränderlichen Audit-Log statt über riskante nachträgliche Datenbankeingriffe.
